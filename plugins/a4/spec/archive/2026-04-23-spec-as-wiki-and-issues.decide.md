# Decision Record: Spec as Wiki + Issues

> Source: design conversation extending the 2026-04-23_2119 handoff (`a4/INDEX dashboard` redesign, second half)

## Context

The previous session (`.handoff/2026-04-23_2119_a4-rename-and-compass-index-dashboard.md`) landed the first half of a document-management redesign — the `a4/INDEX.md` dashboard regenerated by compass, plus the `a4/` lowercase rename. Its named next candidate was "spark lifecycle writes" (candidate 1 in the handoff). When that work started, a status-field conflict surfaced (`status: draft | final` for document state vs. `status: open | promoted | discarded` for spark lifecycle), which prompted a step back into the underlying document model.

From that step back, the user redirected to a different angle:

> "frontmatter를 적극적으로 활용하는 방안으로. use case는 case 마다 파일 하나. review도 review 적용이 필요한 사항마다 파일 하나로. spec을 git issue나 jira issue처럼 관리하는 방향으로."

The consolidated mental model the conversation arrived at:

> **"wiki + github issue를 markdown으로 구현하는 것."**

That is, `a4/` is a git-native implementation of a wiki (narrative pages describing the project) plus an issue tracker (actionable items with lifecycle), with Obsidian dataview serving as the human-facing rendering layer.

## Success Criteria

Ranked by importance:

1. **Per-item lifecycle is first-class** — each UC, task, review finding, and decision has an independent status, updated timestamp, labels. Answer "what's open?" without reading prose.
2. **Cross-cutting narrative has a home** — domain model, architecture overview, actor definitions, context framing do not become an "overview.md" catchall; each concern is its own wiki page.
3. **Derived views never duplicate source data** — UC diagram, authorization matrix, issue lists, etc. are rendered from source (via Obsidian dataview) rather than maintained as separate files.
4. **Git-native** — everything is plain markdown + YAML frontmatter. No database, no external service. Tool-agnostic fallback view (plain tables in INDEX.md) is always available.
5. **No artifact homelessness** — every artifact type (open item, question, review finding, decision, etc.) has exactly one canonical location. No ambiguity about where to write.

## Decision

### The wiki + issue duality

| GitHub concept | Our implementation | Location |
|----------------|--------------------|----------|
| **Wiki page** — "this is how the project is shaped" | `context.md`, `domain.md`, `architecture.md`, `actors.md`, `nfr.md`, `plan.md` | `a4/` root |
| **Issue** — "this needs to be addressed" | UC, task, review item | `a4/usecase/`, `a4/task/`, `a4/review/` |
| **Closed issue archive / ADR** — "this is how we decided" | decision records | `a4/decision/` |
| **Issue tracker dashboard** | `INDEX.md` (+ dataview queries) | `a4/INDEX.md` |
| **Labels, milestones, relations** | frontmatter fields | per-issue |
| **Cross-references (`#123`)** | wikilinks (`[[UC-3]]`) or relative paths | body and frontmatter |
| **Comments** | append-only `## Log` section | per-issue |

### Directory layout

`a4/` is a **single-workspace** — typically one topic per `a4/`. Topic name does not appear in filenames or folders. If a repo genuinely needs multiple disparate topics, split into separate `a4/` instances or introduce topic prefix as an exception.

```
a4/
  context.md                            # Wiki: original idea, problem framing, success criteria
  domain.md                             # Wiki: concepts + relationships + state transitions
  architecture.md                       # Wiki: stack, components, interfaces, test strategy
  actors.md                             # Wiki: actor metadata (type, role, description)
  nfr.md                                # Wiki: non-functional requirements (omit if none)
  plan.md                               # Wiki: implementation strategy, milestones, sequencing rationale
  bootstrap.md                          # Wiki: bootstrap report (latest snapshot; TBD per-run archiving)

  usecase/
    1-share-summary.md                  # Issue: Use Case
    2-search-history.md
  task/
    3-render-markdown.md                # Issue: Task (Jira sense — unit of executable work)
    4-parse-config.md
  review/
    5-missing-error-handling.md         # Issue: review item (finding | gap | question)
    6-add-validation.md                 # stage-scoped reviews also live here; target: frontmatter
    7-concurrency-semantics.md
  decision/
    8-caching-strategy.md               # Issue (resolved): ADR

  spark/
    2026-04-12-1530-vscode-ext.decide.md
  archive/
    ...

  INDEX.md                              # dashboard (regenerated by compass)
```

Conventions:

- **Items go in folders.** Whenever an artifact type can have multiple instances (UC, task, review item, decision), it lives in its own folder.
- **Wiki pages are flat.** Each cross-cutting concern (context / domain / architecture / actors / nfr / plan / bootstrap) is a single file at `a4/` root.
- **Plan is a wiki page, not a folder.** `plan.md` holds the cross-cutting implementation narrative (strategy, milestones, sequencing). Executable work units are tasks in `task/`. The earlier folder name `plan/` for per-item files was renamed to `task/` because each file corresponds to a Jira "task" — a unit of work, not the plan itself.
- **All reviews consolidated in `review/`.** No stage-scoped review subfolders. Stage association is expressed via the `target:` frontmatter field (e.g., `target: usecase/3-search-history`). Cross-cutting gaps/questions omit `target:` or set it to a wiki page.
- **File naming: `<id>-<slug>.md`.** Folder indicates type (no `uc-`/`task-`/`rev-`/`d-` prefix). **Id is a monotonically increasing integer global to the workspace** — unique across all issue types in a given `a4/` (same semantics as GitHub issue numbers). Next id is `max(existing ids in a4/) + 1`. Cross-folder references use full path (e.g., `[[task/3-render-markdown]]`) for readability; basename alone (`[[3-render-markdown]]`) is also unambiguous because ids are unique.
- **No topic prefix.** Filenames do not encode topic (e.g., just `1-share-summary.md`, not `my-app.1-share-summary.md`).
- **Obsidian markdown throughout.** Body uses Obsidian wikilinks (`[[task/3-render-markdown]]`) and embeds (`![[task/3-render-markdown]]`). Frontmatter paths are plain strings (no brackets) for dataview compatibility.

### Unification of open items, questions, and review findings

All three collapse into the **review item** file type. They differ only in origin, which is a frontmatter field:

| Kind | Meaning | Example |
|------|---------|---------|
| `finding` | Reviewer agent flagged it | `usecase-reviewer-r2` found missing validation |
| `gap` | Spec author flagged a known incompleteness | "error handling for UC-3 not defined yet" |
| `question` | Unresolved question that blocks progress | "how should concurrency be handled?" |

They all live in the `review/` folder (or the stage's `review-*.md` subset), share the same frontmatter schema, same lifecycle (`open | in-progress | resolved | dismissed`), and are queryable together via dataview.

This eliminates the current "Open Items / Open Questions / review-report.md" triplet that blurs the distinction.

### Derived views — no files, rendered on demand

The following artifacts are **derived from source and must not exist as hand-maintained files**:

- **Use Case Diagram** — rendered from UC frontmatter (`actors`, `depends_on`, `labels`) + titles into a PlantUML block.
- **Authorization Matrix** — assembled from `UC × actors × permissions` in UC frontmatter.
- **Use Case Relationships / Groups** — dataview query over `depends_on` / `blocks` / `labels`.
- **Next Steps** — dataview filter on `review/*.md` where `status == "open"` sorted by `priority`.

Rendering is **Obsidian dataview** for interactive human use. For plain-text fallback (git web viewer, terminal), compass regenerates a static markdown approximation into `INDEX.md` on each invocation.

### Milestones

Milestones are included as a first-class concept. Frontmatter field `milestone: <name>` on items groups issues into deliverable sets (e.g., `v1.0`, `beta`, `phase-1`). INDEX can group open items by milestone, and dataview can render milestone progress (X of Y resolved).

`assignee` is **not** adopted — the a4 pipeline assumes single-user workflow.

Issue templates (`.github/ISSUE_TEMPLATE/`-style) are **not** adopted — skills create files programmatically, so the skill code IS the template.

### Frontmatter schema

Ids are **monotonically increasing integers global to the workspace** (same semantics as GitHub issue numbers — unique across all issue types). Next id is `max(existing ids in a4/) + 1`. Cross-folder references use full path (e.g., `usecase/3-search-history`, no `.md` extension — Obsidian basename resolve) for readability. Relationship fields store only the **forward direction**; reverse directions (`blocks`, `implemented_by`, `justifies`, `superseded_by`, `children`) are derived by dataview queries.

#### Structural relationships (hybrid)

Structural relationships (that drive lifecycle, blocking, or dashboard queries) are declared in frontmatter with typed fields. Soft references (mentions, see-also) are expressed as Obsidian wikilinks in body prose and surfaced via Obsidian backlinks — no frontmatter maintenance required.

| Field | Direction | Meaning |
|-------|-----------|---------|
| `depends_on` | forward | This item needs the referenced items completed first (lifecycle blocker) |
| `implements` | task → UC | Task delivers these UC(s) |
| `target` | review → any | What this review item is about |
| `wiki_impact` | review/issue → wiki | Wiki pages that need update when this item resolves |
| `justified_by` | any → decision | Decision(s) that justify this item |
| `supersedes` | decision → decision | This decision replaces the referenced decision(s) |
| `parent` | any → any (same type) | Parent item in decomposition hierarchy |
| `related` | any → any | Generic catchall for ties that don't fit other fields but warrant frontmatter-level searchability |

Omit fields that are empty; `[]` is also fine.

#### Per-type schemas

**Wiki page (`context.md`, `domain.md`, etc.)** — minimal, no lifecycle:
```yaml
kind: context | domain | architecture | actors | nfr | plan | bootstrap
updated: 2026-04-24
```
Wiki pages do not declare outbound relationships in frontmatter; their cross-references live as Obsidian wikilinks in body prose + the `## Changes` footnote section (see Wiki update protocol below).

**`plan.md` structure.** Holds: Overview (one paragraph), Implementation Strategy (approach, constraints), and a `## Milestones` section with one `###` subsection per milestone (goal, scope as task wikilinks, success criteria, risks). Milestone scope is declared by the `milestone:` field on each task — `plan.md` narrates; dataview or wikilinks surface the membership. A single file scales to most projects; per-milestone splits are out of scope (see Rejected Alternatives).

**Use Case (`usecase/<n>-<slug>.md`):**
```yaml
id: 3
title: Search history
status: draft | implementing | done | blocked
actors: [user, admin]
depends_on: [usecase/1-share-summary]
justified_by: [decision/2-caching-strategy]
related: [task/5-render-markdown]
labels: [search, ui]
milestone: v1.0
created: 2026-04-20
updated: 2026-04-24
```

**Task (`task/<n>-<slug>.md`)** — Jira "task" semantics: a unit of executable work that implements one or more UCs. Plan-level narrative (strategy, milestone sequencing) lives in `plan.md`, not here.
```yaml
id: 5
title: Render markdown
status: pending | implementing | complete | failing
implements: [usecase/3-search-history, usecase/4-render-preview]
depends_on: [task/4-parse-config]
justified_by: [decision/2-caching-strategy]
files: [src/render.ts, src/render.test.ts]
cycle: 1
labels: [renderer]
milestone: v1.0
created: 2026-04-22
updated: 2026-04-24
```

**Review item (`review/<n>-<slug>.md`):**
```yaml
id: 6
kind: finding | gap | question
status: open | in-progress | resolved | dismissed
target: usecase/3-search-history         # may be omitted for cross-cutting
source: usecase-reviewer-r2 | self | drift-detector
wiki_impact: [architecture, domain]      # basenames; empty/omitted when no wiki update needed
priority: high | medium | low
labels: [validation]
milestone: v1.0
created: 2026-04-24
updated: 2026-04-24
```
The body typically embeds the target for visual context: `![[usecase/3-search-history]]`.

**Decision (`decision/<n>-<slug>.md`):**
```yaml
id: 2
title: Caching strategy
status: final
framework: weighted-scoring
decision: "LRU in-process cache with TTL=300s"
supersedes: [decision/1-no-cache]
related: [usecase/3-search-history]
tags: [caching, performance]
created: 2026-04-24
```

Exact lifecycle vocabularies per type, date typing (YAML-native for dataview), and comment/log section format for issues are deferred to a future schema-tuning round.

### Wiki update protocol

Wiki pages carry no lifecycle but are continuously updated. Updates are driven by **related issue state changes** (create / status transition / resolve), not by calendar time. All wiki edits flow through a single unified conduit: **review items**.

#### Change tracking via footnotes

Modified wiki sections carry sequential footnote markers (`[^1]`, `[^2]`, …) inline. A `## Changes` section at the bottom of each wiki page resolves each marker with date + Obsidian wikilink to the causing item:

```markdown
## Caching

LRU cache with TTL=300s.[^1]
Cache invalidates on logout.[^2]

## Changes

[^1]: 2026-04-23 — [[decision/2-caching-strategy]]
[^2]: 2026-04-22 — [[usecase/3-logout-behavior]]
```

Footnote labels are **local sequential numbers** (file-scoped). The payload is an Obsidian wikilink pointing to the causing issue — this carries both the human-readable "why" and the machine-parseable cross-reference. Multiple causes in one section are expressed by multiple markers side-by-side (`[^1][^2]`). Granularity (sentence / paragraph / section) is situational.

Entries in `## Changes` are kept when they still carry meaning. Older entries that no longer aid understanding may be pruned — git history remains the full audit.

#### Review item as unified conduit

All wiki updates originate from a review item. Three entry paths all converge on `wiki_impact`:

| Scenario | Flow |
|----------|------|
| **Single issue edit (active skill session)** | Skill performs the issue edit. Because the skill has already loaded relevant wiki context to understand the edit, it emits an **in-situ nudge** asking whether a wiki update is needed. If yes, a review item is opened and resolved within the same flow. |
| **Reviewer agent output** | Reviewer creates N review items; each carries `wiki_impact` in frontmatter if the reviewer detected a wiki-affecting gap. **No nudge during the reviewer run** (would be noisy). Surfaces to the user later via INDEX or when picking up the review item for resolution. |
| **Bulk generation (auto-bootstrap, large automated writes)** | Skill calls the **drift detector** as its final step. Detector emits review items with `wiki_impact` set. Required because per-edit nudging does not scale to batch writes. |

**Nudge trigger.** Significant changes only — creation, status transition, resolution. Minor edits (typo, metadata-only) do not nudge. Skill judgment.

**Close guard.** A review item with non-empty `wiki_impact` cannot transition to `resolved` until each referenced wiki page contains at least one footnote whose payload wikilinks the causing issue. Check is performed at close time. Enforcement is a **warning with override** (`"architecture has no footnote for [[review/6-...]], resolve anyway?"`) — user retains final say.

**Footnote labeling at resolve time.** When a review item resolves by editing wiki, the new footnote payload is the **original causing issue** (e.g., `[[usecase/3-search-history]]` or `[[decision/2-caching-strategy]]`), not the review item id. The review item is the conduit; the footnote records the substantive cause.

#### Drift detector

A shared detector scans wiki files and cross-checks footnote references against current issue state. Invocation sites:

- **Embedded in bulk-generation skills** — e.g., `auto-bootstrap` calls it as its final step before returning.
- **Compass Step 3** — every compass invocation runs drift detection as part of gap diagnosis; catches accumulated drift between sessions.
- **Optional `/drift`** — manual invocation mid-session if the user has done a large batch of edits and wants an immediate check.

Output is always **review items** (with `wiki_impact` set and `source: drift-detector`), feeding back into the unified path above.

## Rejected Alternatives

| Option | Reason |
|--------|--------|
| Keep aggregated stage files (`<topic>.usecase.md` with all UCs in one file) | Per-item lifecycle, per-item labels, per-item dataview queries all become awkward. The original pain point (reviews and open items buried in a narrative file) persists. |
| Per-topic subdirectory (`a4/<topic>/usecase/uc-*.md`) | User rejected as unnecessary complexity — `a4/` is already per-project, and multi-topic is handled by separate `a4/` instances or, rarely, by filename prefix. |
| Topic prefix in filenames (`<topic>.uc-1.md`) | Redundant when `a4/` is already single-workspace. Adds noise. |
| Single `overview.md` for all cross-cutting narrative | Collapses distinct concerns (domain model, architecture, context) back into a catchall, defeating the wiki-page model. |
| Separate `open/` and `question/` folders alongside `review/` | All three are "things that need attention on the spec"; separating by origin adds folder overhead without semantic benefit. Unified under `review/` with a `kind` field. |
| `origin: spark/<filename>` field in usecase frontmatter | Redundant — `sources: [{file, sha}]` already serves this, and compass's spark→pipeline link uses `promoted` on the spark side. |
| `archived: true` frontmatter field | Previously rejected in the 2026-04-23_2119 session; folder location (`a4/archive/`) is the flag. Reaffirmed here. |
| "Next action" column in INDEX | Previously rejected; INDEX is an alert board, per-topic narrative stays in compass Step 3. Reaffirmed. |
| GitHub `assignee` field | Single-user workflow; no meaning. |
| Issue templates as files | Skills generate files programmatically; skill code is the template. |
| Type prefix in filenames (`uc-`, `task-`, `rev-`, `d-`) | Folder already indicates type; prefix is redundant noise. Filenames become `<id>-<slug>.md`. |
| Folder-scoped ids (`usecase/1`, `task/1`, `review/1` coexisting) | Ambiguity when ids are referenced by number alone; conflates type with identity. Global monotonic ids (GitHub-issue semantics) are unique across the workspace, so basename references are unambiguous and filename collisions cannot happen. |
| Active wiki updates (skill auto-edits wiki on every issue change) | Too invasive; risks clobbering human edits and makes the skill responsible for far-reaching content decisions. Nudge-based flow preserves human judgment. |
| Drift-detection-only (no in-situ nudge) | Forces the detector to reload wiki context in a separate run; wasteful because the skill that just edited the issue already had wiki context loaded. Nudge leverages that loaded context at near-zero cost. |
| Date-based staleness trigger for wiki updates | Misaligned — wiki freshness depends on whether related issues have changed, not calendar time. A wiki page untouched for 180 days may be perfectly current if no related issues moved. |
| Footnote label as issue id (e.g., `[^uc-3]`) | Couples footnote label syntax to id scheme and forces unique labels per file-wide cause. Labels as local sequential numbers with wikilink payload in the `## Changes` section is simpler and survives id scheme changes. |
| Explicit bidirectional relationship fields (`blocks` stored alongside `depends_on`, `implemented_by` alongside `implements`) | Doubles maintenance and risks drift between forward/reverse. Store only forward direction; dataview computes reverse on demand. |
| Generic `related: [{kind, ref}]` as the only relationship mechanism | Loses per-field semantic clarity; dataview queries become awkward (`WHERE contains(related, {kind: 'implements', ...})`). Hybrid approach (typed fields for structural relationships + generic `related` catchall + body wikilinks for soft references) is cleaner. |
| Systematic migration tool for existing aggregated files | Legacy content in `visual-claude/A4/` and this plugin's prior `a4/` stays as-is; the new model applies only to new workspaces and new items going forward. Building a one-shot splitter + id reassigner is not worth the engineering cost given that legacy material is bounded and stable. |
| `plan/` folder for per-item work files | The folder name `plan/` was misleading: each file corresponds to a Jira **task** (unit of executable work), not to the plan itself. The plan is a cross-cutting narrative (strategy, milestones, sequencing) that belongs in a single `plan.md` wiki page at root. Folder renamed to `task/`; narrative lives in `plan.md`. |
| Separate `design.md` wiki page alongside `architecture.md` | `architecture.md` already covers stack, components, interfaces, and test strategy — traditional "design" content. Adding `design.md` creates a boundary that is impossible to maintain cleanly (tech stack choice, component decomposition, interface definition all become ambiguous). If `architecture.md` ever grows unwieldy, the right move is a focused split by concern via a new ADR, not a generic `design.md`. |
| Per-milestone plan files (`milestone/v1.md`, `milestone/v2.md`, …) | After tasks are split out to `task/*.md`, the remaining plan narrative (overview + strategy + per-milestone scope + sequencing) is thin — typically 150–250 lines for a medium project. Cross-milestone sequencing rationale (e.g., "phase 3 builds on phase 1 foundation") is the core value of the plan document and flows naturally in one file. Splitting fragments that rationale. Revisit only if a project grows past ~10 milestones or a specific milestone accumulates substantial standalone narrative — handle that via a follow-up ADR, not preemptively. |
| Separate `phase:` frontmatter field on tasks | `milestone:` already serves this role — a phase is a kind of milestone (a named deliverable set with a completion criterion). Introducing a second field doubles the grouping axis and forces every dataview query and every skill to choose one. Phase = milestone. |
| Prefix-based skill naming (`a4-handoff`) to coexist with the global `/handoff` skill | Plugin namespace already separates scopes. The plugin skill is invoked as `/a4:handoff` — a dedicated namespace that cannot collide with the global `/handoff`. Encoding the plugin name inside the skill slug duplicates namespace information and makes cross-plugin reading harder. Skill is named `handoff` inside the `a4` plugin; disambiguation is handled by the namespace at invocation time. *(Principle extended by the skill-naming-convention ADR: the full 3-axis naming system and the `think` → `a4` plugin rename are documented there.)* |

<summary>Continuation: wiki update protocol, file naming, relationships (2026-04-24)</summary>

**Why continue.** The initial decision said "wiki pages have no lifecycle" but did not cover "wiki pages still need to be updated." This round filled that gap and tightened several schema details.

**Update trigger.** Not time-based. Triggered by related **issue state changes** (create / status transition / resolve) plus reviewer-flagged or drift-detected gaps. Date-based staleness was rejected as misaligned (a wiki untouched for 180 days may be perfectly current if related issues are stable).

**Footnote scheme for wiki changes.** Inline sequential markers `[^1]`, `[^2]` tag modified sections; `## Changes` section at file bottom resolves each to `YYYY-MM-DD — [[causing-issue]]`. Labels are local sequential numbers (simple, survives id scheme changes); the wikilink payload carries the substantive cause. Granularity situational, multiple causes per section expressed as `[^1][^2]`.

**Review item as unified conduit.** Rejected both "active (skill auto-edits wiki)" and "passive detector only". The insight: a skill editing an issue has already loaded wiki context to make its edit — asking an in-situ nudge is near-free, while a separate detector would reload the same context. So: skill nudges, user decides, review item opens and resolves in the same flow.

**Three entry paths, one data shape.** Single edit (in-situ nudge), reviewer agent (review items with `wiki_impact`, no mid-run nudge), bulk generation (skill invokes drift detector as final step). All converge on review items with `wiki_impact` frontmatter.

**Close guard.** A review item with non-empty `wiki_impact` warns on close if the referenced wiki pages lack a footnote pointing back to the causing issue. Warning + override (user retains final say).

**Drift detector.** Shared detection logic, invoked from (a) bulk-generation skills as their final step, (b) compass Step 3 each session, (c) optional `/drift` for mid-session manual check. Output is always review items — feeds back into the unified path.

**File naming shift.** `uc-1.share-summary.md` → `1-share-summary.md`. Folder indicates type, so type prefix is redundant. Files are `<id>-<slug>.md`. Stage-scoped reviews consolidated into `review/` folder; stage association via `target:` frontmatter (no more `usecase/review-*.md`).

**Global monotonic ids.** Ids are integers unique across the entire `a4/` workspace (GitHub issue semantics). Next id is `max(existing ids in a4/) + 1`. Implication: filename basename is globally unique, so `[[3-search-history]]` resolves unambiguously even without folder prefix — though path-prefixed form (`[[usecase/3-search-history]]`) is preferred for readability.

**Obsidian markdown throughout.** Body uses Obsidian wikilinks (`[[...]]`) and embeds (`![[...]]`). Review item bodies typically embed their target (`![[usecase/3-search-history]]`) for visual context. Frontmatter paths are plain strings (no brackets, no extension) for dataview compatibility.

**Relationships: hybrid approach.** Structural relationships (drive lifecycle / blocking / dashboards) declared in typed frontmatter fields: `depends_on`, `implements`, `target`, `wiki_impact`, `justified_by`, `supersedes`, `parent`, `related`. Soft references (see-also, mentions) live as Obsidian wikilinks in body prose + Obsidian backlinks — no frontmatter maintenance. Only the **forward direction** is stored; reverse (`blocks`, `implemented_by`, `justifies`, `superseded_by`, `children`) is computed by dataview.

**Legacy.** Existing aggregated files (`visual-claude/A4/`, this plugin's prior `a4/`) stay as-is; no migration tooling. New model applies only to new workspaces and new items going forward.

**Decisions deferred.** YAML grammar for path references (plain string vs bracketed), issue comment/log section format, and exact per-type lifecycle vocabularies (though sketched) remain for follow-up rounds.

<summary>Continuation: plan vs task distinction, plan.md as wiki, milestone as phase (2026-04-24, second round)</summary>

**Why continue.** While beginning implementation (id allocator), the user flagged that the ADR conflated "plan" (the implementation roadmap document) with "task" (a unit of executable work, Jira sense). The previous structure had `plan/<id>-<slug>.md` per-item files, but the file kind was actually Jira-task-shaped (Implementation Unit with `implements`, `files`, `cycle`, acceptance criteria), not plan-shaped.

**Reference case.** `visual-claude/a4/terminal-markdown-renderer.plan.md` — a 1214-line aggregated plan file with 16 IUs. After extracting IUs to per-file tasks, the remaining narrative (Overview, Technology Stack, Implementation Strategy, Phase sequencing) is ~150–250 lines. That narrative is the real "plan" artifact and has no home in the prior ADR.

**Folder rename.** `plan/` → `task/`. The files in that folder are Jira tasks — units of executable work with lifecycle. Nothing else about them changes (frontmatter, relationships, `cycle` field all stay).

**New wiki page: `plan.md`.** Single flat file at `a4/` root (same tier as `context.md`, `architecture.md`, etc.). Contains Overview, Implementation Strategy, and a `## Milestones` section with one `###` subsection per milestone. Milestone scope is declared by the `milestone:` field on each task; `plan.md` narrates sequencing and rationale, dataview or wikilinks surface the membership.

**Why not `design.md`.** Already rejected — `architecture.md` covers traditional design content (stack, components, interfaces, test strategy). A separate `design.md` would force a boundary that cannot be maintained cleanly.

**Why not per-milestone plan files.** After task extraction, plan narrative is thin (150–250 lines). Cross-milestone sequencing ("phase 3 builds on phase 1 foundation") is the plan's core value and flows naturally in one file. Splitting fragments that rationale. Revisit only if a project exceeds ~10 milestones or a milestone accumulates substantial standalone narrative.

**Phase vs milestone.** `milestone:` field already carries phase semantics — a phase is a named deliverable set with a completion criterion, which is exactly what a milestone is. No separate `phase:` field.

**Wiki pages now total 7.** `context`, `domain`, `architecture`, `actors`, `nfr`, `plan`, `bootstrap`. Wiki page `kind:` enum updated accordingly. `plan.md` participates in the wiki update protocol: task state changes or milestone re-sequencing trigger footnote additions and `## Changes` entries, resolved through review items like any other wiki edit.
