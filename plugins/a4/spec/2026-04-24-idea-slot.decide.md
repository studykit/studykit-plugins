# Decision Record: Idea Slot

> Source: gap surfaced while reviewing the `a4-redesign` handoff thread ŌĆö the 2026-04-23_2119 handoff explicitly parked an "Inbox slot design" item, which was never picked up in the subsequent 13 ADR Next Steps nor in the thread-closure handoff. This ADR resolves that parked item.

## Context

The 2026-04-23_2119 handoff listed three items as "parked, not dropped":

> 3. **Inbox slot design.** Whether to adopt callgraph-service's `ideas.md` file pattern verbatim, add a `a4/inbox.md` with the same `active | promoted | discarded` schema, and how compass's INDEX surfaces it (likely a top-most "Inbox (N active)" section). Parked explicitly in this session.
> 4. **Experiments / PoC slot.**
> 5. **DECISIONS.md / ADR layer.**

Item 5 was implicitly resolved by the 2026-04-23 spec-as-wiki-and-issues ADR (via the `decision/` folder). Items 3 and 4 were not revisited. This ADR resolves item 3.

The triggering pain: the a4 workspace has **wiki pages** (coarse, cross-cutting narrative), **issues** (id-bearing lifecycle items ŌĆö usecase/task/review/decision), and **spark** files (session-scoped brainstorm and decide documents). None of these accommodate the "head-just-had-a-thought, want to dump it in 30 seconds" input mode. `spark-brainstorm` produces a full templated session document with Context / Discussion Journey / themed Ideas sections ŌĆö the ceremony is incompatible with a one-liner capture.

## Success Criteria

Ranked:

1. **Quick capture must feel like 30 seconds** ŌĆö the friction gap that made inbox-style pain real in the first place. A skill invocation `/a4:idea <line>` that handles id + slug + frontmatter is the mechanism.
2. **Must not overlap with existing issue types** ŌĆö especially `review/`, which has the closest semantic proximity. The boundary must be explicit and unambiguous.
3. **Fits the Jira-issue data shape** ŌĆö per-item file with id, title, status. Ideas are first-class issues, just a different *kind* of issue (Jira's "Idea / Suggestion" type).
4. **Minimal schema, not a dumping ground** ŌĆö only fields that actually drive lifecycle or queries. No `priority`, no `source`, no `target` (ideas are independent).
5. **Graduation path is explicit** ŌĆö promoted ideas record where they went via `promoted: [<path>]`, same mechanism as `spark/brainstorm`.

## Decision

### `a4/idea/` as a fifth issue-type folder

Layout addition:

```
a4/
  usecase/<id>-<slug>.md
  task/<id>-<slug>.md
  review/<id>-<slug>.md
  decision/<id>-<slug>.md
  idea/<id>-<slug>.md             # NEW
```

Ideas are issues ŌĆö same file structure, same id-allocation policy (monotonic global integer), same archive convention (`git mv` to `a4/archive/` when terminal).

### Boundary with `review/`

The boundary is drawn on **dependency to current progress**, not content:

| Dimension | `review/` | `idea/` |
|-----------|-----------|---------|
| Semantic | "Something is missing / wrong / unclear **in the current spec**" | "Something **independent of the current spec** might be worth doing someday" |
| Progress dependency | Usually blocks progress (`kind: gap` / `question`) or records a flag on existing work (`kind: finding`) | Never blocks. Opt-in to pursue later. |
| `target:` field | Usually present (points at the spec item being reviewed) | **Not in the schema.** Ideas are independent. |
| Provenance | Reviewer agents, drift-detector, self-authored gap | User input, mostly. Captured raw. |
| Lifecycle | `open \| in-progress \| resolved \| dismissed` (you work on it) | `open \| promoted \| discarded` (you don't work on the idea itself ŌĆö you graduate it to usecase/task/spark) |

Judgment rule: "If ignoring this blocks or degrades current spec work, it is a review item. If not, it is an idea."

### Frontmatter schema

```yaml
---
id: 12
title: <one-line title>
status: open              # open | promoted | discarded
promoted: []              # paths populated when status ŌåÆ promoted
related: []               # soft links to other artifacts
labels: []                # free-form tags
created: 2026-04-24
updated: 2026-04-24
---
```

Required: `id`, `title`, `status`, `created`, `updated`.
Optional: `promoted`, `related`, `labels`.

Deliberately **excluded** fields (rationale in Rejected Alternatives):

- `priority` ŌĆö ideas are pre-prioritization. Prioritization belongs to the graduation target (usecase/task).
- `source` ŌĆö ideas are almost always user-authored; source has low information content here.
- `target` ŌĆö ideas are independent of other artifacts by definition.
- `kind` ŌĆö only one kind of idea.
- `milestone` ŌĆö not scheduled.

### Body structure

Free-form. No required sections.

Suggested optional shape for non-trivial ideas:

```markdown
# <title>

<one to three paragraphs describing the idea>

## Why this matters
<optional>

## Notes
<optional>
```

For one-liner captures (the common case via `/a4:idea`), the body may be just the title echoed as H1 and no more.

The skill:

1. Resolves `<project-root>` via `git rev-parse --show-toplevel`; aborts if `a4/` missing.
2. Allocates next id via `scripts/allocate_id.py <project-root>/a4`.
3. Generates a URL-friendly slug from the argument (ASCII lowercase, hyphen-joined; Korean characters pass through).
4. Writes `a4/idea/<id>-<slug>.md` with the schema above, `status: open`, today as `created`/`updated`, argument as `title` and H1.
5. Reports the new path.

`disable-model-invocation: true` ŌĆö user must type `/a4:idea` explicitly.

### Promotion flow

Promotion is **manual today**. When an idea graduates:

1. Open the idea file.
2. Change `status: open` ŌåÆ `status: promoted`.
3. Set `promoted: [<path-to-new-artifact>]`.
4. Bump `updated:`.
5. Create the target artifact separately (new usecase, task, spark session, etc.).

A potential `/a4:idea-promote <id>` skill is out of scope for this ADR ŌĆö revisit when repeated manual promotion becomes a pain point.

Supported promotion targets (convention, not enforced):

- `spark/<basename>` (brainstorm or decide) ŌĆö for ideas worth exploring or deciding on in depth.
- `usecase/<id>-<slug>` ŌĆö for ideas that map directly to a requirement.
- `task/<id>-<slug>` ŌĆö for ideas that are concrete executable work.

Explicitly NOT a promotion target: `review/` (that would invert the boundary drawn above).

### Discarded ideas

`status: discarded` + `updated:` bump. Body may add a one-line `## Why discarded` if the reason is worth preserving. Archive via `git mv` to `a4/archive/` at the user's discretion.

### INDEX.md surfacing

New section `## Open ideas (N)` in `a4/INDEX.md`, rendered by `scripts/index_refresh.py`. Includes a dataview block (live Obsidian rendering) and a static fallback list of open idea titles with wikilinks. Section appears after `## Spark (open N)` as it is the other pre-pipeline-capture surface. Section is skipped entirely when no ideas exist.

### Validator

`validate_frontmatter.py` gains an `idea` schema:

- Required: `id`, `title`, `status`, `created`, `updated`
- `status` enum: `{open, promoted, discarded}`
- `id` must be int; date fields `YYYY-MM-DD`
- `promoted` and `related` validated as path-list fields (plain strings, no brackets, no `.md`)
- `idea` added to `ISSUE_FOLDERS` so id-uniqueness check includes ideas
- Body validator (`validate_body.py`) needs no change ŌĆö idea files follow the same wikilink-resolution rules as other issues.

## Rejected Alternatives

| Option | Reason |
|--------|--------|
| **Single `a4/inbox.md` file with line-entry accumulation** (callgraph-service pattern verbatim) | Considered and discussed. User chose Jira-issue-style per-item files for consistency with other issue types and for enabling dataview queries / per-item lifecycle. The friction-gap argument against per-item was addressed by the `/a4:idea` skill absorbing id + slug + frontmatter generation so the user still only types one line. |
| **Reuse `review/` with new `kind: idea`** | Collapses two semantically distinct concepts. Review lifecycle (`open \| in-progress \| resolved \| dismissed`) implies "you work on it"; idea lifecycle (`open \| promoted \| discarded`) implies "you graduate or drop it". Review items are bound to current progress via `target:`; ideas are independent. Review items are queried alongside drift alerts in INDEX; ideas belong in a different surface. |
| **Folder name `inbox/`** | `inbox` is a container metaphor (entries accumulate inside). Per-item files are better named by the Jira issue-type convention the other four folders already follow: usecase/task/review/decision are all **singular issue-type nouns**. `idea/` fits that pattern; `inbox/` does not. |
| **Include `priority` field** | Ideas are pre-prioritization ŌĆö the point of capture is to not yet decide. Prioritization attaches to the graduation target (usecase/task's existing `priority` / backlog ordering). Keeping `priority` on idea would just create a second knob that has to be kept in sync. |
| **Include `source` field** | Low information content ŌĆö ideas are almost always user-authored directly or forwarded from user-adjacent inputs. The `source` field on review items distinguishes `self` / `drift-detector` / `<reviewer-agent-name>` which *are* meaningfully different. For ideas, everything effectively collapses to `self`. If a future flow needs to distinguish (e.g., ideas promoted from external sources like customer feedback), add the field then. |
| **Include `target` field** | Would blur the boundary with `review/`. If an input is best expressed as "this needs attention on item X", it is a review item, not an idea. |
| **Include `kind` field** | Only one kind of idea. Adding `kind: idea` would be tautological with the folder name. Review's `kind` exists because three distinct kinds (finding / gap / question) share one folder; idea has no such split. |
| **Enforce a body template** | Ideas are captured in 30 seconds. Any required structured body defeats the capture-friction goal. Optional shape is documented as a convention; no validator rule enforces it. |
| **Require-and-record `discarded_reason:`** | Considered for parity with spark-brainstorm's `discarded_reason` (which that skill never writes in practice ŌĆö it is a documented field with no writer). Ideas are cheaper to discard than brainstorms; requiring a reason would add friction on the discard path, which is the second-most common terminal transition. A body `## Why discarded` is suggested but optional. |
| **Put ideas in `a4/spark/` alongside brainstorm/decide** | Spark files are session-documents (one file per facilitated session). Idea files are line-documents (one file per captured thought). Mixing granularities in one folder would make `a4/spark/` schizophrenic. The "pre-pipeline capture" proximity is real, but orthogonality of session vs. line granularity is the stronger constraint. |
| **`/a4:idea` with interactive prompts (no argument form)** | Would introduce a multi-turn dialogue for a single-line capture. The whole point is 30-second friction. Single-argument form is the default; interactive mode is unnecessary. If the user types `/a4:idea` with no argument, the skill politely reminds them to provide a line. |
| **Auto-commit the new idea file** | Every other a4 skill leaves its output in the working tree for the user to commit. Consistent behavior is better than special-casing idea. |
| **Auto-promote detection (spot when an idea's title matches a new usecase title)** | Premature. Re-examine after real usage reveals whether this is a common pattern or an edge case. |
| **Separate experiments-slot in the same ADR** | Item 4 from the 2026-04-23_2119 parked list is distinct ŌĆö experiments are execution-phase artifacts (temporary investigations, PoCs), not pre-capture thoughts. Conflating would dilute both. Experiments remain parked; a future ADR may pick them up. |
