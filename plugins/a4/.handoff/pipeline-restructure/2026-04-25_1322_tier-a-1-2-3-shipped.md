## What changed in design vs. the previous handoff

- `/a4:run` step numbering is **flat 1–5**, not 2.1–2.5. The previous handoff said "split Phase 2 out … Step 2.5". I renumbered when the file became its own skill. `references/frontmatter-schema.md` was updated to match (UC `shipped` writer is now "/a4:run Step 5"; task `implementing` / `complete` writer is "/a4:run Step 2"; task `failing` writer is "/a4:run Step 2 / 3").
- `/a4:run` reads `roadmap.md` Launch & Verify primarily; **falls back to `bootstrap.md` § Launch & Verify** for UC-less projects with no roadmap. **Halts** if neither exists. The final fallback for projects that decline both is unresolved (Tier C item 12 above).
- Workflow-mode hook integration is **in-process via `import`**, not `uv run` subprocess. `a4_hook.py session-start` calls `_workflow_mode_sweep()` then `_workflow_mode_init()` (sweep first so a fresh init does not collide with a stale orphan). `a4_hook.py session-end` calls `_workflow_mode_cleanup()`. All three helpers swallow exceptions silently per hook conventions. The `init` helper is idempotent — it leaves already-existing state untouched rather than calling the CLI's `cmd_init` (which would `die(code=2)` on collision).
- The `SessionEnd` hook entry is **additive**: the existing `bash cleanup-edited-a4.sh` stays for the edited-record concern; the new `a4_hook.py session-end` runs alongside it for the workflow-mode concern. They are parallel, lifecycle-symmetric, separate concerns.

## How `/a4:run` behaves today

Frontmatter:

```yaml
default_mode: autonomous
mode_transitions:
  to_conversational:
    - task-implementer returns clarification_needed or ambiguous decision
    - test-runner failure requires task / roadmap / arch / UC classification
    - UC ship-review (Step 5) — explicit user confirmation per UC
    - 3-cycle cap reached with failures still outstanding
    - destructive operation proposed
  to_autonomous:
    - skill invoked directly (`/a4:run`)
    - resume after Step 4 classification once user confirms the corrective edit
    - resume after Step 5 ship-review once user has confirmed every candidate UC
```

Step list:

1. **Pick Ready Tasks.** Ready when `status ∈ {pending, failing}`, every `depends_on` resolves to `complete`, and either `implements:` is non-empty with referenced UCs at `ready` / `implementing`, **or** `implements:` is empty (UC-less ready conditions vacuously pass).
2. **Spawn task-implementer.** One agent per ready task. `transition_status.py` flips `pending → implementing` before spawn, `→ complete` / `→ failing` after.
3. **Run Integration + Smoke Tests** via `test-runner`. Per-failure review items at `target: task/<id>` (or `target: roadmap` when not task-traceable).
4. **Analyze Results.** Transition to `conversational` for classification. Branches: task/roadmap → revise + reset, arch → stop + recommend `/a4:arch iterate`, usecase → stop + recommend `/a4:usecase iterate`. 3-cycle cap halts with `failing` tasks.
5. **UC ship-review (conditional).** Skipped entirely when no in-scope task has non-empty `implements:` (UC-less normal case). Otherwise, per-candidate two-to-four-sentence verdict, user confirmation, `transition_status.py` flips UC `implementing → shipped` (cascade flips `supersedes:` targets to `superseded`).

## How `/a4:task` behaves today

Frontmatter: `default_mode: conversational`, with mode_transitions surfacing kind ambiguity, `implements:` / `justified_by:` ambiguity, AC-source ambiguity (e.g., `feature` with neither UC nor ADR pointer), and spike-sidecar confirmation.

Eight-step body: pre-flight → capture intent → resolve `implements:` / `justified_by:` → compose body → propose `spike/<id>-<slug>/` (kind: spike only) → allocate id + write file → refresh `implemented_by:` (skip when `implements:` empty) → spike sidecar create (if confirmed) → hand-off to `/a4:run`.

The skill never spawns implementation agents. Initial file write is at `status: pending`; subsequent transitions go through `transition_status.py` from `/a4:run`.

## How `a4_hook.py` is wired today

Subcommands (in dispatch order in `main()`): `post-edit`, `stop`, `session-start`, `session-end`. Each subcommand local-imports what it needs to keep the post-edit fast path lean.

Session-start ordering:

```
sweep workflow-mode state files older than 1 day
init workflow-mode state at "conversational" if absent
refresh implemented_by (existing)
report status-consistency (existing)
emit JSON payload if there is anything to say
```

Session-end:

```
cleanup workflow-mode state file for this session_id
return rc=0
```

The `bash cleanup-edited-a4.sh` and `bash sweep-old-edited-a4.sh` are unchanged — they handle the parallel edited-record lifecycle and run alongside the Python dispatcher.

## A small accident, fully recovered

While smoke-testing the hooks I ran `rm -rf a4` from the studykit-plugins root to clean up an `a4/` directory my test had created. macOS APFS is case-insensitive by default, so the command **also matched the existing `A4/` directory** (24 archived markdown files in `A4/archive/` plus `A4/issues.yml`). I caught it on the next `git status --short` and ran `git checkout HEAD -- A4/` — everything tracked is back. The only loss was an empty untracked `A4/spark/` directory. Lesson logged: never `rm -rf a4` (lowercase) from this repo's root.

The next session should not be surprised to see `A4/` (uppercase) at the project root. It is a separate workspace from the plugin's own `plugins/a4/` and is unrelated to this restructure work.
